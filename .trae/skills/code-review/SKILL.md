# Code Review

对代码进行**全方位审查**，覆盖以下 7 大维度。审查结果分级输出：Critical / Warning / Suggestion。

---

## 审查流程

1. **扫描范围确认** — 确认审查的文件或目录范围
2. **逐文件分析** — 按 7 大维度逐一检查
3. **问题分级** — 按严重程度标记：🔴 Critical → 🟡 Warning → 🔵 Suggestion
4. **输出报告** — 结构化输出，标明文件、行号、问题描述、修复建议

---

## 七大审查维度

### 1. 🔒 安全漏洞 (Security)

参考 OWASP Top 10，重点检查：

| 检查项 | 说明 |
|--------|------|
| **注入攻击** | SQL 注入、NoSQL 注入、命令注入、LDAP 注入 |
| **XSS** | 反射型/存储型/DOM 型 XSS，未转义的用户输入 |
| **CSRF** | 跨站请求伪造，敏感操作缺乏 Token 校验 |
| **敏感信息泄露** | 密钥/AK/SK/Token/密码硬编码；日志中打印敏感数据；`.env` 文件提交 |
| **认证与授权** | 弱密码策略、Token 无过期时间、权限校验缺失或可绕过 |
| **不安全的反序列化** | `eval()`、`unserialize()` 等危险函数 |
| **依赖安全** | 使用已知漏洞版本的第三方库 |
| **文件操作** | 路径穿越、任意文件读取/上传、未限制文件类型 |
| **加密** | 使用弱加密算法(MD5/SHA1用于密码)、硬编码盐值 |

### 2. 🐛 Bug 检测 (Bugs)

| 检查项 | 说明 |
|--------|------|
| **空值/未定义引用** | 未判空直接访问属性或方法 (NPE / undefined) |
| **边界条件** | 数组越界、除零、整数溢出、空集合处理 |
| **异常处理** | 空 catch、吞异常、finally 中 return、异常信息泄露内部细节 |
| **逻辑错误** | 条件判断反了、`=` 写成 `==`、死循环、不可达代码 |
| **并发问题** | 竞态条件、非原子操作、死锁风险 |
| **资源泄漏** | 未关闭的连接/文件句柄/数据库连接 |
| **类型错误** | 隐式类型转换导致意外结果 |
| **异步陷阱** | Promise 未 await、回调地狱、未处理的 rejection |

### 3. ⚡ 性能问题 (Performance)

| 检查项 | 说明 |
|--------|------|
| **N+1 查询** | 循环内查询数据库/调用接口 |
| **不必要计算** | 循环内重复计算不变值、重复请求相同数据 |
| **内存问题** | 大对象未及时释放、闭包导致的内存泄漏、事件监听未移除 |
| **前端渲染** | 大列表未使用虚拟滚动、频繁 DOM 操作、未防抖/节流 |
| **资源加载** | 未压缩图片、未使用 CDN、未启用缓存、未按需加载 |
| **数据库** | 缺失索引、`SELECT *`、未分页 |
| **冗余网络请求** | 重复请求、未合并请求 |

### 4. 📐 代码规范 (Style & Standards)

| 检查项 | 说明 |
|--------|------|
| **命名规范** | 变量/函数/类命名是否清晰、符合项目约定（camelCase/PascalCase/snake_case） |
| **代码格式** | 缩进一致、行长度、括号风格、空行使用 |
| **注释质量** | 无注释的复杂逻辑、过时/误导性注释、被注释掉的代码 |
| **文件组织** | 导入顺序混乱、文件过长（>500行）、循环依赖 |
| **类型使用** | TypeScript 项目：滥用 `any`、缺少类型定义、类型不精确 |
| **ESLint/TSLint** | 是否违反项目配置的 lint 规则 |

### 5. 🏗️ 可维护性 (Maintainability)

| 检查项 | 说明 |
|--------|------|
| **DRY 原则** | 重复代码块，应抽取为公共函数/组件 |
| **单一职责** | 函数/类承担了过多职责（函数 > 50 行需关注） |
| **魔法数字/字符串** | 未定义为常量的硬编码值 |
| **耦合度** | 模块间过度依赖、循环依赖 |
| **圈复杂度** | 嵌套过深（>3 层）、过多的 if-else/switch |
| **副作用** | 函数修改全局状态、非纯函数 |

### 6. 🧪 可测试性 (Testability)

| 检查项 | 说明 |
|--------|------|
| **依赖注入** | 是否便于 mock？硬编码依赖导致难以测试 |
| **全局状态** | 直接依赖 `window`/`localStorage`/全局变量，难以隔离测试 |
| **测试缺失** | 新增功能是否缺少对应测试 |
| **函数可测性** | 函数是否过于复杂难以编写单元测试 |

### 7. 🎨 架构设计 (Architecture)

| 检查项 | 说明 |
|--------|------|
| **分层清晰** | 业务逻辑是否混入 UI 层/数据访问层 |
| **组件粒度** | 单文件组件是否过大、职责不清 |
| **状态管理** | Pinia/Vuex store 是否合理拆分、跨组件通信是否合理 |
| **API 设计** | 接口设计是否 RESTful、参数/返回值是否合理 |
| **路由设计** | 路由嵌套是否合理、权限控制是否到位 |
| **扩展性** | 修改现有功能是否需要大量改动（开闭原则） |

| # | 文件:行号 | 维度 | 问题 | 修复建议 |
|---|----------|------|------|---------|

- Critical: X 个

- Suggestion: X 个

## 注意事项

- 审查当前项目使用的技术栈：Vue 3 + TypeScript + Vite + Pinia
- 遵循项目已有的编码风格和目录结构约定
- 安全审查需要特别关注用户输入、数据库操作、文件操作
- 对于不确定的问题，标记为 Warning 而非 Critical
- 提供可执行的修复代码，而非仅仅描述问题
